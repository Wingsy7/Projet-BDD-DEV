NOTES PROJET ECOLE AVANCE

1. IDEE DE DEPART

Le projet regroupe 2 parties :

- une base de donnees MySQL sur une ecole
- une API Python avec un menu admin

On a voulu garder le meme theme partout pour que la base, l'API et le menu
soient relies entre eux.

2. PAR QUOI ON A COMMENCE

On a commence par la base.

On a d'abord pose les tables principales :

- specialite
- promotion
- cours
- prof
- eleve
- dossier
- absence
- note

Ensuite on a ajoute les tables de liaison quand c'etait utile :

- prof_cours
- eleve_cours

3. POURQUOI CETTE STRUCTURE

Le but etait d'avoir une base assez simple a relire, mais assez complete pour
faire les requetes du sujet sans tricher.

Par exemple :

- un eleve appartient a une promotion
- une promotion appartient a une specialite
- un cours peut etre commun ou rattache a une specialite
- un eleve peut suivre plusieurs cours
- un prof peut enseigner plusieurs cours

4. DONNEES DE DEPART

On a mis un vrai jeu de donnees pour que les resultats soient plus parlants.

On a ajoute :

- plusieurs promos
- plusieurs profs
- plus d'eleves que dans la premiere version
- des absences differentes
- des notes fortes, moyennes et faibles
- des entreprises et des contrats d'alternance

Comme ca, les tris, moyennes, sous-requetes et regroupements donnent des
resultats qu'on peut vraiment commenter.

5. REQUETES SQL

Une fois la base faite, on a ecrit les requetes du sujet :

- SELECT simples
- jointures
- AVG, SUM, GROUP BY, HAVING, ORDER BY
- sous-requetes
- insert, update, delete

On a fait cette partie avant l'API pour verifier que la base etait deja
coherente.

6. PROCEDURE ET TRIGGERS

On a ajoute une procedure de classement des eleves.

Le score tient compte :

- de la moyenne
- des absences
- des avertissements

On a aussi ajoute les triggers demandes :

- creation automatique du dossier
- ajout de l'info "mineur" si l'eleve a moins de 18 ans
- garder la meilleure note
- mettre une note a 10 si le prof est supprime

7. API

Quand la base etait stable, on a fait l'API avec FastAPI.

L'idee etait de garder des requetes SQL simples dans `requetes.py`, puis de
faire les traitements dans `traitements.py`.

Exemples :

- bonnes notes
- profs severes
- heures d'absence
- regroupement des notes

8. MENU ADMIN

Le menu admin a ete fait apres l'API.

Il ne parle pas directement a MySQL.
Il passe par l'API avec `requests`.

Le menu reste simple :

- lire
- creer
- modifier
- supprimer

Le menu affiche aussi les listes avant de demander un id, ce qui evite de taper
des valeurs au hasard.

9. PARTIES EN PLUS

On a garde 2 ajouts :

- les clubs
- l'alternance

Les clubs servent surtout a montrer une autre relation plusieurs-a-plusieurs.
L'alternance permet d'avoir des entreprises et des contrats eleve/entreprise.

10. CE QUI ETAIT LE PLUS DELICAT

Les points les plus delicats etaient :

- faire une base assez riche sans la rendre illisible
- respecter le format exact de certaines routes
- faire cohabiter les triggers sans bug
- garder un menu simple

11. ORDRE DE TRAVAIL

L'ordre final a ete :

1. penser les tables
2. faire la base
3. charger des donnees
4. faire les requetes SQL
5. ajouter la procedure et les triggers
6. faire l'API
7. faire le menu admin
8. verifier l'ensemble

12. CE QU'ON RETIENT

Ce projet nous a surtout servi a relier plusieurs choses vues en cours :

- modeliser une base
- faire des requetes SQL utiles
- recuperer les donnees en Python
- retravailler les resultats en Python
- exposer la base avec une API
- utiliser cette API dans un petit outil admin
